## Code Canonicality

Same logic must have exactly one implementation in the repo. Version evolution belongs in git history, not in filenames or identifiers.

- Before adding any function, package, module, or file: grep for similar implementations first.
- Forbidden suffixes for filenames and identifiers: `_v1`, `_v2`, `_v3`, `_new`, `_old`, `_backup`, `_temp`, `_copy`, `_final`, `_real`, `_improved`, `_refactored`, `_fixed`, `_legacy`, `_deprecated`.
- Refactor in place. Git history is the safety net; do not keep old and new implementations side by side.
- No commented-out code, dead exports, scratchpad directories, or parallel transition implementations.
- Do not commit `tmp/`, `scratch/`, `backup/`, `_old/`, `deprecated/`, `archive/`, or `wip/`.

## Project Identity

`agentsdk-go` is a Go Agent SDK that implements core Claude Code-style runtime capabilities, plus middleware interception, hooks, MCP, sandboxing, skills, subagents, and examples for CLI/HTTP/enterprise integration.

- **Primary consumers**: Go developers building agent runtimes, CLIs, CI/CD integrations, and enterprise agent platforms.
- **Lifecycle**: maintained public SDK/library.
- **Team scale**: public OSS.
- **AI tools targeting this repo**: Claude Code, Cursor, Aider, Droid, Codex CLI, and future coding agents.

### Dependency Policy

- `go.sum` is committed and must stay in sync with `go.mod`.
- Dependabot checks Go modules and GitHub Actions weekly.
- New dependency PRs require a justification line: what it does and why existing dependencies are insufficient.
- Major dependency upgrades require human review.
- Minimum release age policy: prefer dependencies older than 3 days unless a security fix requires faster adoption.
- Allowed licenses by default: MIT, Apache-2.0, BSD-3-Clause, ISC.

### Go-Specific Rules

- Errors are values. Wrap cross-package errors with `fmt.Errorf("...: %w", err)`.
- No `panic` in library code except impossible invariant violations that are documented.
- No `os.Exit` outside `cmd/`.
- Avoid package-level mutable state.
- Use `context.Context` for cancellation and goroutine lifecycles.
- Tests live as `*_test.go`, preferably table-driven. Use `t.Parallel()` where safe.
- Integration tests that require external services or API keys must not be part of the default offline gate.

### Module Boundaries

- Treat `docs/refactor/PRD.md` and `docs/refactor/ARCHITECTURE-v2.md` as ground truth.
- `pkg/model` exposes the single `model.Model` abstraction; providers must adapt to it, not fork it.
- `pkg/sandbox` is isolation only. It must not grow approval or permission workflow responsibilities.
- Runtime orchestration belongs in `pkg/api`; lower packages should not import `pkg/api`.
- Examples may depend on public SDK APIs; SDK packages must not depend on examples.

## Important Development Notes

1. Keep `pkg/sandbox` as isolation only; do not add approval workflows there.
2. Keep `pkg/model` centered on one `model.Model` interface; do not add parallel provider-specific runtime contracts.
3. Do not place online examples requiring API keys in default offline checks.
4. Runtime loads `./AGENTS.md` with `@include` support and appends it under `## Memory`; keep this file concise and operational.

## Conventions

### Naming

- Go files use standard lowercase Go naming with underscores only for test files and conventional multiword filenames.
- Tests use `*_test.go`.
- Package names are short, lowercase, and stable.
- Forbidden canonicality suffixes are listed in `## Code Canonicality`.

### Commit Format

- Conventional commits required: `<type>(<scope>): <message>`.
- Allowed types: `feat`, `fix`, `refactor`, `chore`, `docs`, `test`, `ci`, `build`, `perf`.
- Scopes should match repo areas: `api`, `model`, `tool`, `middleware`, `hooks`, `sandbox`, `config`, `message`, `mcp`, `runtime`, `cli`, `docs`, `ci`.

### Pull Requests

- Use `.github/PULL_REQUEST_TEMPLATE.md`.
- Required sections: Summary, Test plan, Risk, Public API impact.
- For public API changes, call out whether the change is breaking.
- For new dependencies, justify the dependency.
- For CI/release workflow edits, call them out explicitly.

### Branch and Release Model

- Trunk-based development on protected `main`.
- Agents may work on feature branches, never push directly to `main`.
- Releases are tag-based: pushing `v*` tags triggers GitHub Release workflow.
- Conventional commits are required for readable history, not for automatic semantic release.

### Coverage

- Target line coverage: 70%.
- Current policy: report-only. CI uploads/prints coverage, but the full-repo threshold is not yet a hard failing gate.

## Code Review Self-Check

Before requesting review, agents must verify:

- The change matches existing package boundaries and patterns.
- Similar code was searched for and reused/refactored where appropriate.
- `make check` was run locally or the skipped part is explicitly explained.
- Tests were added or updated for non-trivial behavior changes.
- Public API changes are called out in the PR.
- No `_v2`, `_new`, `_temp`, scratchpad, dead export, or commented-out-code drift was introduced.
- The diff is focused and does not include unrelated cleanup.

## Architecture Discipline

- Architecture source of truth: `docs/refactor/ARCHITECTURE-v2.md`.
- Requirements source of truth: `docs/refactor/PRD.md`.
- This repo uses layered package boundaries, not strict DDD bounded contexts.
- TDD is preferred: for non-trivial changes, start with or update a failing behavior test when practical. If not practical, explain the validation path in the PR.
- Public interface changes are governed by tests plus PR review. No API snapshot tool is currently required.

## Agent Operating Rules

### Agents May

- Edit code and tests on feature branches.
- Run local validation commands.
- Update `AGENTS.md` when conventions materially change, with human review.
- Open PRs targeting `main`.

### Agents Must Not

- Push directly to `main`.
- Force-push shared branches.
- Use `--no-verify`.
- Publish releases or push release tags.
- Run production deploys.
- Persist API keys, real user data, private endpoints, tokens, `.agents/settings.local.json`, or sensitive local trace/log content.
- Add major dependency upgrades without human review.
- Modify CI/CD files without explicitly flagging the change in the PR.

### Commit Attribution

Agent-authored commits should include co-authorship when supported:

```text
Co-authored-by: <agent name> <agent email or noreply address>
```
